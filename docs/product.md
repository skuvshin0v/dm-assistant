# dm-assistant — Product Document

## Суть продукта

**Единое место для мира кампании + ИИ, который знает этот мир.**

DM один раз записывает информацию — ИИ помнит её всегда, помогает найти, генерирует идеи в логике кампании.

---

## Целевая аудитория

**Приоритет MVP:** Домашние мастера (ведут локальные группы для друзей).
**Следующий сегмент:** Профессиональные мастера (зарабатывают на ведении).

**Паттерн боли (подтверждён кастдевами):**

1. Данные разбросаны — Notion + Obsidian + Figma + Telegram + Excel одновременно
2. Кампания не помнит себя — continuity теряется между сессиями, имена NPC вылетают из головы
3. ИИ не знает вайб — ChatGPT/GPT4 дают топорные идеи без контекста кампании

---

## Wow-моменты (в порядке приоритета)

1. **"Само запоминает"** — вставил заметки в чат → AI разобрал сущности → предложил создать NPC и локации → подтвердил → мир наполнен
2. **"Знает мой мир"** — написал "кто такой Варрен?" → AI ответил точно, с деталями из заметок
3. **"Идеи в стиле моего мира"** — попросил придумать квест → получил идеи, органичные для своей кампании

**Целевой онбординг (5 минут):**

```
Зарегистрировался
→ Создал мир и кампанию
→ Вставил заметки в чат [Записать]
→ AI предложил сущности → подтвердил
→ Спросил "кто такой Варрен?"
→ AI ответил точно
→ 💡 "Оно помнит мой мир"
```

## Модель данных

### Иерархия

```
User
  └── World
        └── Campaign
              └── Chat (несколько на кампанию)
                    └── Message
              └── Session (постфактум)

Entity (принадлежит World, шарится между всеми кампаниями мира)
Source (текстовый артефакт из чата)
Idea (сохранённые идеи из браинсторма)
```

Все сущности (Character, Location, Faction, Item) живут на уровне **мира**. Кампания добавляет только историю: чаты, сессии, Sources. Никакого разделения "мировые vs кампанийные" сущности нет.

**Чат на уровне мира:** Chat с `campaign_id=null` — контекст = сущности мира без привязки к конкретной кампании. Для лора и браинсторма по миру в целом.

### Сущности (canon layer)

| Сущность        | Описание                                              | MVP                               |
| --------------- | ----------------------------------------------------- | --------------------------------- |
| **Character**   | Именованный индивид: NPC или PC                       | ✅                                |
| **Location**    | Именованное место: от плана бытия до комнаты          | ✅                                |
| **Faction**     | Коллективный агент: организация, гильдия, государство | ✅                                |
| **Item**        | Именованный лорный объект (не инвентарь, не механика) | ✅                                |
| **Event**       | Атомарное изменение состояния мира или действие       | ✅ (упрощённый, отдельный подход) |
| **StoryThread** | Сюжетная линия, агрегирует события                    | ❌ v2 (отдельный подход)          |

**Правило:** нет имени — нет сущности. Игровые механики (HP, уровень, статы) не хранятся.

### Видимость

Два уровня: `public` (дефолт) и `gm_only`.

Visibility работает на двух уровнях независимо:

**1. Уровень сущности** — вся карточка скрыта от игроков.
- Тайный злодей, скрытая локация, секретный план фракции → entity `gm_only`

**2. Уровень поля / факта** — сущность игрокам известна, но отдельные факты скрыты.

Основные поля сущности (name, description, status) = всегда DM-правда. Per-fact visibility реализуется через `key_facts: list[{content, visibility}]`:

```
Character "Варрен":
  status: dead                          ← DM знает
  status_visibility: gm_only
  key_facts:
    - {content: "капитан стражи",       visibility: public}   ← игроки видят
    - {content: "притворяется живым",   visibility: gm_only}  ← только DM
```

Игроки видят только `public` key_facts. Поля с `gm_only` для них скрыты.

**Source с DM-заметками** ("на самом деле Варрен — лич") → Source `visibility: gm_only`.

**На MVP:** entity-level visibility (`public` / `gm_only`). Хранится, отображается бейджем в UI. RECORD автоматически проставляет `gm_only` при маркерах секретности в тексте ("на самом деле", "только мастер знает", "скрыто от игроков").

**v2:** per-field visibility через `key_facts: list[{content, visibility}]`. Player-facing view фильтрует `gm_only` сущности и факты.

### Source

Source — сырой текст из чата, помеченный как "запись". Не равен Message.

```
Source:
  world_id        (всегда)
  campaign_id     (nullable — лор без кампании)
  chat_id         (nullable)
  message_id      (nullable — ссылка на исходное сообщение)
  session_id      (nullable — если привязан к сессии)
  kind            (session_summary | npc_notes | location_notes | lore | misc)
  content         (text)
  visibility
```

В будущем: `kind=file` для загрузки документов. Схема не сломается.

**Правило:** Source создаётся только через чат (chat-first). Никаких кнопок "загрузить" на MVP.

### Session

Session — лёгкая постфактум-сущность. Создаётся после игры, не до.

```
Session:
  campaign_id
  number         (порядковый, монотонный)
  title          (nullable)
  played_at      (nullable)
  summary        (nullable — пишет мастер или генерирует AI)
```

**Создание — два пути:**

- Кнопка "Записать сессию" → создаёт Session → открывает контекст записи
- Через чат: мастер вставляет заметки → RECORD детектит маркеры сессии → предлагает "Создать сессию 7?" → подтверждение

Для подготовки к сессии — обычный чат (браинсторм), Session не нужна.

### Idea

```
Idea:
  chat_id
  content
  source_message_id  (откуда пришла идея)
  status             (new | saved | used | discarded)
  created_at
```

Мастер может сохранять понравившиеся идеи из браинсторма. Список сохранённых идей на кампанию.

## AI Pipeline

### Режимы (MVP: явные кнопки в UI)

Три кнопки над чатом: **Записать / Найти / Идеи**

Мастер выбирает режим явно. Авто-классификатор интента — финальный полиш после того, как все три вертикали устойчивы (v2).

Одно сообщение может активировать несколько режимов — агент обрабатывает их в нужном порядке сам.

### Агентный подход

Вместо фиксированного порядка — single agentic loop:

```
Message + режим → Agent(message, available_tools)
→ Agent решает порядок вызовов
→ Готовит результат для подтверждения мастером
→ Мастер подтверждает → execute
```

LLM только предлагает. Мастер — финальный арбитр всех изменений.

### RECORD (Записать)

```
Текст
→ fuzzy/vector search по именам сущностей в БД → top-N кандидатов со скорами
→ LLM(текст + кандидаты) → предлагает действия:
    [update Character "Варрен": добавить "предатель гильдии"]
    [create Location "Пещера Теней"]
    [create Session 7 из заметок]
→ UI показывает предложения → мастер подтверждает/отклоняет → execute
→ создать Source
```

Один LLM-вызов. LLM не создаёт/обновляет напрямую — только предлагает.

### Merge логика

Сущность в базе = всегда актуальное состояние. Event = история изменений.

**Два сценария:**

**1. Additive** — нового факта нет в базе для этого поля.
→ UI: 🟢 "Добавить к Варрену: носит красный плащ" → подтвердить

**2. Conflict** — новое значение противоречит существующему.
→ UI: 🔴 diff "статус: мёртв → жив" → мастер выбирает что актуально → обновить

LLM никогда не резолвит конфликты автоматически. Только предлагает — мастер решает.

Severity (мягкое обновление vs критическое противоречие) влияет только на цвет в UI (🟡 vs 🔴), флоу одинаковый.

### SEARCH (Найти)

```
Запрос
→ vector search по entities → inject топ-N релевантных сущностей
→ LLM формирует ответ в стиле кампании
```

### BRAINSTORM (Идеи)

```
Запрос
→ load context: релевантные сущности + последние сессии
→ LLM генерирует идеи с учётом мира и сеттинга
→ UI предлагает сохранить понравившиеся → Idea entity
```

## Freemium

**Принцип:** один платящий юзер покрывает ~100 бесплатных (основные затраты — токены).

| Тариф | Лимиты                                     | Логика                                        |
| ----- | ------------------------------------------ | --------------------------------------------- |
| Free  | 30 сообщений/мес, 1 кампания, 50 сущностей | Получить магию → упереться → конвертироваться |
| Paid  | 500 сообщений/мес, ∞ кампании, ∞ сущности  | Активный мастер: ~2-3 сессии в месяц          |

**Правило:** лимит срабатывает ПОСЛЕ wow-момента, не до.

Детализация лимитов и ценообразование — уточняется после первых реальных юзеров.

## Валидация

MVP не выходит в публичный релиз до итерации 9. Валидация — личные созвоны с 3-5 знакомыми DM-ами.

| После итерации | Что показываем | Вопрос |
| --- | --- | --- |
| 3 | RECORD: вставить заметки с реальной сессии → подтвердить предложения | "Сработало как ожидал? Что сломалось?" |
| 5 | Все три режима: Записать / Найти / Идеи | "Использовал бы регулярно? За деньги?" |

Один созвон = один DM, одна реальная кампания, без скрипта. Слушаем где застрял.

## Виральность

**Шаринг сущности** — кнопка "Поделиться" на карточке Character / Location / Faction.

Генерирует публичную ссылку на read-only view карточки. DM кидает в Discord другому DM: "вот мой злодей". Приходящий видит карточку + CTA "создай своего".

Встроен в итерацию 4. Хранение: `entity.is_public` + `entity.public_slug`.

## Итерации

Каждая итерация — один рабочий сеанс с осязаемым результатом.

---

### Итерация 1 — "Скелет с базой"

**Цель:** минимальный рабочий продукт — лендинг, авторизация, структура данных, навигация.

**Что делаем:**

- Лендинг / посадочная страница (до авторизации): что такое продукт, CTA "Начать"
- Supabase: схема (worlds, campaigns, chats, messages, entities, sessions, sources, ideas) + RLS на всех таблицах
- Auth: email+password
- UI: создать мир → создать кампанию → список чатов → открыть чат (поле ввода, сообщения не отправляются)
- Раздел "Канон кампании" — заглушка "раздел пока пустой"

**Осязаемый результат:** открыл лендинг → зарегистрировался → создал мир → создал кампанию → вижу чат и Канон.

---

### Итерация 2 — "Само запоминает" ⭐ WOW #1

**Цель:** рабочий RECORD pipeline.

**Что делаем:**

- FastAPI backend: `/record` endpoint
- fuzzy search кандидатов из БД (trigram/BM25; vector search — итерация 5)
- LLM(текст + кандидаты) → предлагает действия
- UI подтверждения: список предложений, мастер принимает/отклоняет
- Создание Source при каждой записи
- Авто-детект сессии + создание Session

**Осязаемый результат:** вставляю заметки → AI предлагает NPC и локации → подтверждаю → мир наполнен.

---

### Итерация 3 — "Знает мой мир" ⭐ WOW #2

**Цель:** рабочий SEARCH — AI отвечает с контекстом кампании.

**Что делаем:**

- `/search` endpoint: inject сущностей в контекст (пока все, ограничение по размеру явное)
- UI: режим "Найти", чат с ответами
- Базовая агентная логика: RECORD + SEARCH вместе

**Осязаемый результат:** спрашиваю "кто такой Варрен?" → AI отвечает точно из моих данных.

---

### Итерация 4 — "Идеи в стиле мира" ⭐ WOW #3

**Цель:** рабочий BRAINSTORM.

**Что делаем:**

- `/brainstorm` endpoint: контекст = релевантные сущности + последние сессии
- Системный промпт: "ты знаешь этот мир, генерируй в его логике"
- UI: режим "Идеи", сохранение идей → Idea entity
- Шаринг сущности: `entity.is_public` + `entity.public_slug`, кнопка "Поделиться" на карточке

**Осязаемый результат:** прошу придумать квест → получаю идеи с именами из моего мира.

---

### Итерация 5 — "Точный поиск в большом мире"

**Цель:** RAG вместо inject all.

**Что делаем:**

- pgvector в Supabase: embeddings для сущностей
- Генерация embedding при создании/обновлении сущности
- SEARCH и RECORD используют vector search вместо inject all

**Осязаемый результат:** большие миры (500+ сущностей) работают корректно и быстро.

---

### Итерация 6 — "Кампания помнит историю"

**Цель:** Event как упрощённая сущность.

**Что делаем:**

- Event: `id, campaign_id, session_id (nullable), description, kind (lore/in_session/between_sessions), visibility`
- RECORD распознаёт события в тексте (не только сущности)
- Timeline сессии в UI

**Осязаемый результат:** вижу хронологию событий. AI учитывает историю при ответах.

---

### Итерация 7 — "Авто-режим"

**Цель:** авто-классификатор интента вместо явных кнопок.

**Что делаем:**

- Classifier: быстрая модель возвращает `intents: list`
- Убираем кнопки режимов (или оставляем как override)
- Агент полностью решает порядок

**Осязаемый результат:** пишу в чат без выбора режима — AI сам понимает что делать.

---

### Итерация 8 — "Монетизация включена"

**Цель:** freemium gate, продукт готов к монетизации.

**Что делаем:**

- Счётчик сообщений (per user, per month)
- Лимиты free tier
- UI: прогресс-бар лимита, paywall при достижении
- Подписка (Stripe или альтернатива для CIS)

**Осязаемый результат:** юзер видит лимит, упирается после wow-момента, может подписаться.

---

## Не входит в MVP

- Импорт из Notion/Obsidian
- Загрузка файлов/документов (Source=file — v2)
- StoryThread
- Коллаборация (несколько юзеров в одной кампании)
- Шаринг с игроками (player-facing view)
- Карта мира
- Игровые механики (HP, статы, инвентарь)
- Мобильное приложение
