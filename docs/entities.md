# Концептуальная модель сущностей

---

Определяем для каждой сущности: **что она такое по смыслу, какую задачу мастера решает, где её границы с соседними сущностями.**

Решения по концепции — продуктовые, не технические. Поля и типы из них следуют, не наоборот.

Порядок закрытия (от самой «нагруженной смыслом» к самой утилитарной):

1. **Faction** — закрыта (см. ниже).
2. **Event + StoryThread** — закрыты (см. ниже).
3. **Real-world слой: Session + Source** — закрыт (см. ниже).
4. **Character** — закрыт (см. ниже).
5. **Item / Artifact** — закрыт (см. ниже).
6. **Location** — закрыт (см. ниже).

#### Faction — концептуальная модель _(закрыто 2026-05-12)_

**Определение.** Фракция — это **коллективный агент в мире**: организованная группа NPC с общей идентичностью, целями и/или ресурсами. Действует в мире как один игрок, даже если состоит из ста человек. Отличается от отдельного NPC (один индивид), расы/вида (биология, не выбор) и локации (территория, не агент).

**Задачи мастера, которые решает фракция:**

1. **Источник связных действий в мире** — мастеру не нужно решать «что сделает каждый вор», достаточно «что сделает Гильдия». Снимает когнитивную нагрузку.
2. **Пружина для конфликта и квестов** — фракция = напряжение. Войны/союзы/шпионаж между фракциями = долгоиграющий сюжет.
3. **Контейнер для безымянных NPC** — «один из Стражников Корнхейма» уже несёт контекст (лояльность, манеру, ресурсы) без расписывания индивидуального NPC.
4. **Аккумулятор репутации** — партия спасла Орден → Орден помнит. Репутация копится на фракции, а не на каждом её члене.
5. **Социальный контекст локаций** — «эта территория контролируется Y». Без фракций локации — мёртвые места.
6. **Долгосрочные арки** — месть культу, восхождение в гильдии, война с империей. Фракция — естественный носитель арки (StoryThread часто привязан к фракции).
7. **Идентичность для партии** — PC может быть членом фракции или хотеть им стать. Даёт мастеру рычаг для миссий «от лица фракции» и моральных дилемм.

**Правила «что считать фракцией»:**

- ✅ Имя собственное **И** общая идентичность/цели — это фракция.
- ❌ Безымянная группа («жрецы», «культисты», «контрабандисты», «несколько стражников») — не фракция, даже если описана подробно.
- ❌ Раса / вид (эльфы, орки) — никогда не фракция, биология не выбор. «Совет Эльфийских Старейшин» — фракция; «эльфы» — нет.
- ✅ Семья / династия / род — фракция, **если** есть собственное имя Дома + групповая идентичность (Дом Корвен, Клан Воронов). Просто фамилия не считается.
- ✅ Функциональная группа, привязанная к месту/имени — фракция (Стража Корнхейма, Гарнизон форта Эзра, Жречество Храма Огня). Абстрактное «стражники» — нет.
- ⏸ Партия PC по умолчанию не фракция. Если игроки явно зарегистрировали имя/идентичность партии — становится фракцией. На MVP откладываем.

**Граница с Location и State (Q1):**

Государство — это **Location** (level=state). Правящие структуры, политические партии, армии внутри государства — это отдельные **Faction**'ы. Это разделяет «где это на карте» и «кто принимает решения».

Внутри одного государства может жить много фракций (Корона + Совет Купцов + Гильдия Воров + Орден Алой Розы). Внутри одного города/локации — тоже (Стража + Гильдия + Совет).

**Связи фракции (Q5, Q6, Q7):**

- **Иерархия плоская.** Если две сущности имеют разные имена — это две отдельные Faction'ы, без `parent_faction_id`. Связь «крыло/подразделение» описывается через `relations` со словесным kind.
- **`relations` как явное поле на фракции** — статичные отношения (alliance / hostile / vassal / rival / neutral / member-of) с другими фракциями и с персонажами. Хранятся **один раз, на стороне фракции**; карточка персонажа строится через обратный lookup. Динамика отношений — через Event'ы; relations отражает текущее состояние, а история — в событиях.
- **Множественная принадлежность.** NPC может быть членом нескольких фракций одновременно (Барон Форлек: глава Серого Клинка + член Дома Корвен). Это даёт пищу для конфликтов лояльности.

**Видимость (Q8):**

У фракции есть флаг видимости: `public` (все знают) / `known_to_party` (партия знает) / `gm_only` (только мастер). На MVP это просто хранится, в UI выводится бэйджем; полноценная фильтрация «что видит партия» — Сценарий 4–5 или позже. Но поле закладываем сразу, чтобы не переписывать карточки.

**Открытый вопрос: вражда между государствами.**

Государство = Location, а Location ↔ Location отношений у нас нет. «Варн в войне с Финемером» нужно где-то хранить.

Склоняемся к **fallback с авто-Faction**: для каждого государства существует подразумеваемая правящая Faction. Если мастер её не назвал — extraction создаёт по шаблону («Корона Варна», «Двор Финемера»). Тогда война государств = `relations` между их правящими фракциями. Плюсы: чистая модель, перевороты сохраняют историю, агентный поиск работает («покажи всех союзников Варна»).

Окончательное решение принимаем на стадии правок промпта (Шаг 2 ниже), когда будет понятно, как это уживается с extraction. Само поле `relations` это не блокирует.

#### Event — концептуальная модель _(закрыто 2026-05-12)_

**Определение.** Event — это **факт, что что-то произошло**: атомарное изменение состояния мира или намеренное действие. В отличие от описания (state) сущности: «Барон носит красный плащ» — state, «Барон погиб в битве» — event.

**Задачи мастера, которые решает Event:**

1. **Хронология мира** — «в каком порядке всё было».
2. **История изменений сущности** — «когда Борин стал капитаном». Без event этот факт растворяется в свободном `description`.
3. **Связи сущностей через время** — Партия ↔ Барон ↔ замок связаны через event «партия проникла в замок».
4. **Память для отыгрыша** — NPC ссылается «как тогда в Финемере», мастер поднимает конкретный event.
5. **Базис сюжетных арок** — StoryThread = последовательность связанных events.
6. **Источник причинности** — «почему этот NPC нас ненавидит» → event «партия предала Серый Клинок».
7. **Аналитика игры** — «сколько боёв было», «когда последний раз появлялся культ».

**Правила «что считать event'ом»:**

- ✅ Изменение состояния мира (смерть, рождение, разрушение, перемещение, передача предмета).
- ✅ Намеренное действие персонажей/фракций (атака, договор, побег, предательство, обряд).
- ✅ Открытие/находка/откровение (нашли артефакт, узнали тайну).
- ❌ Просто описание / state (как выглядит локация, кто чем владеет «вообще»).
- ❌ Размышление мастера / OOC.
- ❌ Пассивное присутствие («партия была в городе три дня» без значимых действий).

**Зерно event'а — атомарное.** Одно event = одно атомарное действие/изменение с одним временным маркером. «Проникли в замок и нашли Печать» = два events (находка — следствие проникновения). Композитные события («Битва за Ридден» как зонтик) моделируются как StoryThread, не как один большой event.

**Виды (`kind`):** `lore` (до партии или вне игр), `in_session` (то, что делала партия), `between_sessions` (мир жил между сессиями), `unknown` (fallback). `parallel` дропаем как непонятный.

**Хронология — относительная, не абсолютная.** Систему «реальных дат» не затаскиваем, душно для мастера. Якорь у события — один из:

- **Сессионный** (для `in_session`/`between_sessions`): `session_id` + `order_in_session`. Между сессиями — по `session_number`.
- **Относительный** (для `lore`): `before_event_id` — одиночная ссылка-якорь «это случилось до того event'а». Если в одной заметке два lore-события и одно явно раньше другого — связь явная.
- **Эпоха** (опционально): `era_label` — свободный ярлык эпохи для группировки в UI.

Абсолютная дата (`date_text`, `in_game_date`, `date_confidence`) хранится только если мастер явно написал. Никаких автоматических вычислений «10 лет назад → конкретный год» на MVP.

**Причинность.** Не храним напрямую (отдельных полей `caused_by` / `causes` нет). Причинная цепь = последовательность events в одном StoryThread; «X было раньше в этой ветке, значит Y из-за X». Если в Сценарии 4 окажется недостаточно — добавим.

**Последствия (consequences) — дропаем как поле.** Последствие = либо следующий event (атомарный, kind=between_sessions, идёт в той же StoryThread), либо обновление state самой сущности (Faction.status, Location.controlled_by_faction_id). Свободный текст ничего не обновляет и порождает дубли с описанием.

**Участники.**

- `participant_character_ids` — только именованные персонажи.
- `party_involved: bool` — флаг «партия как агент действовала» (по правилу: action связан хотя бы с одним PC из партии **или** в тексте явно «партия что-то сделала» без перечисления имён). Решает баг «Партия как character_id» из Iter 1.
- `factions_involved: list[str]` — фракции в событии.
- `location_ids: list[str]` — места (список, не одиночное поле: «битва от моста до стен» — два места).
- `item_ids: list[str]` — предметы, фигурирующие в событии.

**Связь Event ↔ StoryThread.** N к N: `Event.story_thread_ids` + `StoryThread.event_ids`. Один event может быть в нескольких ветках («партия убила тролля» = ветка «защита моста» + ветка «отношения с городом»).

**Importance.** Оставляем `minor / normal / key`. Калибровка значимости — задача extraction-промпта в Iter 2.

**Граница event vs state vs упоминание** — серая зона, разруливается в промпте. Правило: event только при значимом изменении состояния или значимом действии. Рутинные перемещения и пустые разговоры — не события.

**Timeline сущности — без новых полей.** «Как менялась сущность во времени» вычисляется обратным lookup: все events, где сущность упомянута, по хронологии. На карточке — секция «История». Diff state (status: alive → dead) — post-MVP.

**Правило: lore, узнанный в сессии = два события.** Если в сессии партия раскрыла факт из истории мира («партия нашла газету: король Малдрен был убит 100 лет назад»), extraction создаёт **два** events:

1. **lore event** — про сам исторический факт. `kind=lore`, `session_id=null`, in_game date примерно фиксируется (через `before_event_id` / `era_label` / `date_text`).
2. **in_session event** — про момент обнаружения этого факта партией. `kind=in_session`, `session_id=current`, `party_involved=true`.

Связь между ними — через общие сущности (оба ссылаются на Character: Малдрен) и/или общий StoryThread. Visibility lore event: `known_to_party` после обнаружения; `gm_only`, если обнаруженная партией версия — заведомо ложная и истинного события партия не знает. Это правило идёт в extraction-промпт Iter 2.

#### StoryThread — концептуальная модель _(закрыто 2026-05-12)_

**Определение.** StoryThread — это **ментальная модель сюжетной линии мира**: агрегированный взгляд на то, как развивается определённая линия сюжета через события, сущности и их связи. Это не квест, не TODO и не план мастера; это способ ткнуть в линию и увидеть, как она разворачивалась в мире.

StoryThread не должен предписывать будущее. В каноне хранится только то, что есть в мире или уже произошло. Планы «что может случиться дальше» живут в `Idea` или будущем Quest workspace, не в StoryThread.

**Задачи мастера, которые решает StoryThread:**

1. **Собрать события в сюжетную линию** — «что происходило по культу Кровавой Паутины» без ручного поиска по всем сессиям.
2. **Показать развитие линии во времени** — какие события были раньше, что было последним, где линия сейчас.
3. **Сшить переплетённые события** — один event может относиться к нескольким линиям; ветки пересекаются через общие events.
4. **Дать карту связанных сущностей** — персонажи, фракции, локации, предметы, которые важны для этой линии.
5. **Помочь мастеру держать ментальную модель сюжета** — не забыть, что «тайна Меча Первых» связана и с Домом Корвен, и с руинами Нар-Талла, и с исчезновением Борина.

**Что StoryThread НЕ является:**

- ❌ Не квест игрока. Квест/подготовка сцены — отдельный Quest workspace.
- ❌ Не статусная задача (`hook`, `stalled`, `resolved`) в таск-трекере.
- ❌ Не план мастера и не future outline.
- ❌ Не большое композитное событие. Композит типа «Битва за Ридден» может быть StoryThread, но тело ветки — отдельные атомарные events.

**Связь с Event.** N к N: `StoryThread.event_ids` + `Event.story_thread_ids`. Хронология ветки вычисляется из хронологии её events. «Первый event ветки» = начало/триггер; отдельное поле `triggered_by_event_id` не нужно.

**Связанные сущности — два уровня:**

- **Неявные** — выводятся из events ветки (участники, фракции, локации, предметы).
- **Явные** — `related_character_ids`, `related_faction_ids`, `related_location_ids`, `related_item_ids`: нужны, когда мастер хочет сказать «эта линия про X» до того, как по X случился event.

**Связи между ветками.** Плоско: `related_thread_ids: list[str]`. Без типизированного kind на MVP. Это означает «эти сюжетные линии переплетаются / влияют друг на друга / тематически связаны». Если в Сценарии 4 понадобится `part_of`, `continues`, `blocks` — расширим.

**Статусы.** Старый `status: hook/active/stalled/resolved/abandoned` выпиливаем: это квестовая модель, она плохо ложится на «ментальную модель сюжета». Вместо этого одно поле `is_open: bool = True`. Закрытая ветка = мастер считает линию завершённой. Никакой детализации «stalled / hook» на MVP.

**Выпиливаем:** `last_progress_event_id` (вычисляется как последний event по хронологии) и `next_steps_hint` (планы будущего должны жить в Idea/Quest, не в каноне).

#### Visibility — общее решение _(закрыто 2026-05-12)_

Поле `visibility` есть на двух уровнях:

```python
Visibility = Literal["public", "gm_only"]
visibility: Visibility = "public"
```

- **На canon-сущностях:** Character, Location, Faction, Event, Item/Artifact, StoryThread.
- **На Source** (real-world слой).

Значения:

- `public` — широко известно / не является секретом.
- `gm_only` — знает только мастер; партия пока не знает.

**Чего НЕТ:**

- Visibility на отдельных полях / key_facts / событиях внутри сущности — не делаем. Сущность для мастера всегда полная.
- Visibility на Session — не делаем (Session = реальная игровая встреча, все игроки в ней участвовали).
- «Двух версий сущности» (для мастера и для игрока) — нет. Сущность одна, фильтрация только на стадии отображения игроку.

**Как работает фильтрация (Сценарий 4+):** игрок не видит:

- Сущности с `visibility=gm_only`.
- Provenance entries, у которых `Source.visibility=gm_only`. Если у сущности **все** provenance из gm_only Source — сущность тоже скрыта.

## На MVP поле просто хранится и показывается бэйджем в UI.

#### Real-world слой: Session + Source _(закрыто 2026-05-12)_

В каноне живёт два слоя сущностей:

1. **Game-world** — Character, Location, Faction, Event, Item, StoryThread. Это сущности выдуманной вселенной.
2. **Real-world** — Session и Source. Это артефакты процесса игры и записи: игровые встречи и текстовые заметки. Они **не часть мира**, но порождают мир (через extracted entities и provenance).

##### Session — игровая встреча в реальном мире

**Определение.** Session — это игровая встреча в реальном мире: «мы играли в четверг 5 мая с 19 до 23». Мост между real-world (когда играли) и game-world (что произошло).

**Жизненный цикл.** Session создаётся **постфактум**, в момент когда мастер пишет в чат заметку про прошедшую игру. До этого Session в каноне не существует ----- это возможно стоит откорректировать

**Создание — только через чат.** На MVP нет UI-кнопки «новая сессия». Чтобы появилась Session, нужен хотя бы минимальный текст в чате (даже «была седьмая сессия, ничего особенного» создаст минимальный Source + Session). Это согласуется с chat-first подходом.

**Источник создания:** extraction в Iter 5 распознаёт маркеры сессии (заголовок «Сессия 4», структура с событиями, метаданные «играли 5 мая») и предлагает в preview «создать Session + Source». Если в одном тексте несколько сессий — баннер `SessionSplitProposal` (уже работает) с кнопкой «создать N сессий».

**Что Session держит (минимум):**

- `number` — порядковый номер встречи **в real-world** (всегда монотонный).
- `kind: SessionKind` — что это за сессия (см. ниже).
- `real_date`, `in_game_start_date`, `in_game_end_date`, `title`, `summary` — опциональные атрибуты.
- `source_ids: list[str]` — какие Source привязаны.

**Не держит:**

- `event_ids` — вычисляется обратным lookup из `Event.session_id`.
- `status` — выпиливаем на MVP.
- `visibility` — не имеет смысла (см. блок Visibility выше).
- `players`, `gm_notes` — пост-MVP. Игроки сессии при необходимости реконструируются ретроспективно из events.

**`SessionKind`** (для нелинейных кампаний):

```
SessionKind = Literal[
    "main",       # сессия в основной хронологии
    "backstory",  # отыгрыш предыстории персонажа / события из далёкого прошлого
    "flashback",  # короткая сцена-флешбэк
    "parallel",   # параллельная ветка (другая партия / другая локация одновременно)
    "oneshot",    # одиночная сессия вне основной хронологии
]
```

Дефолт — `main`. Обязательное поле сессии.

**Два timeline.** `Session.number` и `Session.in_game_start_date` — независимые оси:

- **Real-world timeline** (как мы играли): сортировка events по `session.number → order_in_session`.
- **In-game timeline** (как было в мире): сортировка events по `event.in_game_date` → `event.before_event_id` → `session.in_game_start_date` → `session.number`.

Эти оси могут расходиться — это нормально для backstory/flashback/parallel.

**`in_game_start_date` опциональный.** Если null:

- `kind=main` — сессия идёт сразу после предыдущей main по in-game timeline.
- `kind=backstory/flashback/parallel/oneshot` — попадает в **отдельную ленту «вне основной хронологии»** в UI; точной позиции на timeline нет. Мастер может потом дозаписать дату — сессия встроится.

Никакой обязательной системы дат, мастера не душим. Глубокую проработку timeline UI делаем в Сценарии 4.

**Связь Session ↔ Source.** 1:N. Один Source → 0..1 Session, одна Session → 0..N Source. Через `Source.session_id` (одиночная ссылка) + `Session.source_ids` (обратный список). Только Source с `kind=session_summary` может быть привязан к Session; остальные SourceKind (lore, npc_notes, location_notes, misc) — всегда без Session.

**Связь Event ↔ Session.** Через `Event.session_id`. Правила:

- `kind=in_session` event → `session_id=current session`.
- `kind=between_sessions` event → `session_id=next session` (мастер физически рассказывает о нём в начале следующей сессии).
- `kind=lore` event → `session_id=null`.

##### Source — текстовый артефакт

**Определение.** Source — сырой текст, который мастер написал в чат и пометил «это запись». В отличие от Session (которая знает, что она про игровую встречу), Source семантически нейтрален: может быть саммари сессии, лор-эссе, бэкграунд персонажа.

**MVP-ограничения:**

- Source = **текст only**. Никаких файлов, PDF, голоса, изображений. Это post-MVP.
- Source появляется **только через чат**: мастер пишет → record_planner создаёт Source. Никаких UI-кнопок «загрузить файл/текст».
- Никакого хранения файлов в схеме / на диске.

**Source ≠ Message.** Это разные сущности:

- Одно сообщение может стать несколькими Source (через `SessionSplitProposal`).
- Сообщения, не относящиеся к «записи» (вопросы агенту, брейнсторм), не порождают Source.
- Source должен быть стабильным: если мастер удалит/отредактирует сообщение, Source остаётся. Поэтому `raw_text` копируется в Source при создании.
- У Source своя жизнь и свой id (`src_xxx`).

Добавляем опциональное поле **`source_message_id: str | None`** — ссылка на исходное сообщение (msg_xxx), чтобы из UI можно было «прыгнуть к сообщению-первоисточнику».

**Visibility.** У Source есть `visibility`. NPC-notes / тайные планы / DM-only заметки помечаются `gm_only`. См. блок Visibility выше — фильтрация при показе игроку идёт в том числе через Source.visibility.

**SourceKind.** Оставляем как сейчас: `session_summary / npc_notes / location_notes / lore / misc`. Только `session_summary` может быть привязан к Session.

#### Character — концептуальная модель _(закрыто 2026-05-12)_

**Определение.** Character — это **именованный индивид в мире**: один разумный (или приравненный к разумному) актор с собственной идентичностью. Один человек, один эльф, один дракон, один бог, один лич — каждый из них Character. Индивидуальное место в мире, отдельный голос, отдельные отношения, отдельная история во времени.

Character — не Faction (индивид, а не коллективный агент), не species (не «эльфы», а «Айра-эльфийка»), не «партия» как абстракция.

**Задачи мастера, которые решает Character:**

1. **Реестр того, кого мастер водит** — кого помнить, кого вернуть в сцену, у кого какие связи и мотивы.
2. **Точка отыгрыша** — голос, манера, цели привязаны к индивиду, а не к «жителям города вообще».
3. **Носитель отношений** — кто партии союзник, кто враг, кто нейтрален и почему.
4. **Участник событий** — без Character'ов events абстрактны («кто-то кого-то»).
5. **Носитель состояния во времени** — жив/мёртв, в плену/свободен; через `status` + events.
6. **Реестр PC** — отдельная роль за столом, без которой не отличить «персонажа партии» от «NPC, которого партия знает».

**Что Character НЕ является:**

- ❌ Безымянная группа («трое стражников у ворот»). Стал важен → дайте имя → стал Character.
- ❌ Раса / вид (эльфы, гномы) — это `species` других Character'ов.
- ❌ «Партия» как абстрактное «мы». Каждый PC = отдельный Character; «партия» = `party_involved` на events + опционально Faction.
- ❌ Класс / профессия как сущность. «Кузнецы» — нет; «Хорт-кузнец» — да.

**Q1. Природа существа vs роль за столом vs роль в истории.**

Three поля, не одно:

- **`role: CharacterRole`** — роль за столом: `player`, `npc`, `unknown`.
- **`species: CharacterSpecies`** — биологическая/онтологическая природа: `human`, `elf`, `dwarf`, `halfling`, `gnome`, `orc`, `goblinoid`, `dragon`, `demon`, `deity`, `undead`, `construct`, `spirit`, `beast`, `monster`, `other`, `unknown`. Не свободный текст: extraction должен выбирать из enum, иначе Айра/айрина/эльф-полукровка дают три разных значения.
- **`npc_kind: CharacterNpcKind | None`** — драматическая роль: `commoner`, `ally`, `patron`, `rival`, `villain`, `boss`, `monster`, `deity`, `merchant`, `quest_giver`, `authority`, `other`. Заполняется только для `role=npc`; для `role=player`/`unknown` всегда null.

Примеры: Эллара = `role=npc`, `species=deity`, `npc_kind=patron`. Азрагор = `role=npc`, `species=dragon`, `npc_kind=boss`. Хорт-кузнец = `role=npc`, `species=human`, `npc_kind=commoner`. PC Айра = `role=player`, `species=elf`, `npc_kind=null`.

Если списка не хватит на тестовых прогонах — расширяем enum, не уходим в свободный текст.

**Q2. Безымянные группы — не Character.** «Отряд орков-разведчиков», «команда корабля» без имени и важности живут в `description` event'а или в составе сцены. Если группа становится сюжетно важной и получает имя — становится Faction, не Character.

**Q3. Titles — отдельное поле.** Добавляем `titles: list[str]` — «Барон», «Король», «Сир», «Капитан стражи». `name` хранит только собственное имя («Форлек»). Это нужно, чтобы:

- При апдейте «лишился титула» / «получил новый титул» менять только `titles`, не `name`.
- В UI показывать `Барон Форлек`, а не дублировать титул в имя.
- Уметь группировать «все короли мира».

**UI/search ответственность:** карточка и поиск **должны** строить комбинированные формы. Запрос «барон форлек» обязан матчить Character с `name=Форлек, titles=[Барон]`. Это требование на следующий уровень (карточки/поиск), не на extraction.

**Q4. Локация: «откуда» vs «где сейчас» — два разных поля.**

- **`current_location_id`** — где персонаж сейчас.
- **`origin_location_id`** — откуда родом.

«Хорт-кузнец из Корнхейма, сейчас живёт в Финемере» → `origin=Корнхейм`, `current=Финемер`. Это снимает класс ошибок Iter 1.

`home_location_id` или «обычное место обитания» не добавляем — это уже усложнение, текущей пары хватает на 95% случаев.

**Q5. Статус — минимальный.** `CharacterStatus = "alive" | "dead" | "unknown"`. Всё остальное (без вести пропал, в плену, превращён в нежить, изгнан) — через `key_facts` и `events`. Если в Сценарии 4 покажется тесно — расширим, но не сейчас.

**Q6. Relations: маленький направленный набор + UI строит обратное.**

`relations: list[CharacterRelation]` со структурой:

```
CharacterRelation:
  target_id: str
  kind: CharacterRelationKind
  note: str | None
```

`CharacterRelationKind` стартовый набор:

- Семейные: `parent_of`, `child_of`, `sibling_of`, `spouse_of`, `lover_of`.
- Социальные: `mentor_of`, `student_of`, `friend_of`, `enemy_of`, `rival_of`, `knows`.
- Власть: `serves`, `served_by`, `protects`, `protected_by`.
- Долговые: `owes_debt_to`, `owed_debt_by`.
- Fallback: `other`.

**Правило хранения.** Relation хранится **один раз** на той стороне, из чьей перспективы он был извлечён или подтверждён. Дублирования «зеркальной» relation на другой стороне не делаем (рассинхрон неизбежен).

**Правило отображения.** Карточка персонажа собирает relations из двух источников:

1. Прямые: relations, лежащие на самом Character.
2. Обратные: relations других Character, где `target_id == this.id`, с инверсией `kind`.

Словарь инверсий:

```
parent_of <-> child_of
mentor_of <-> student_of
serves <-> served_by
protects <-> protected_by
owes_debt_to <-> owed_debt_by
sibling_of, spouse_of, lover_of, friend_of, enemy_of, rival_of, knows, other → симметричные (kind не меняется)
```

Тонкие, эпизодические взаимодействия («предал», «спас», «убил») — это **event**, не relation. Relation отвечает на «какая между ними устойчивая связь сейчас», event — «что между ними произошло конкретно».

**Q7. Lorekeeper — не charsheet.**

Зафиксированное правило для extraction-промпта Iter 2:

> Lorekeeper хранит **историю и сюжет**, не игровые механики. Агент не записывает в канон: уровень, HP, AC, статы, навыки, владения, ячейки заклинаний, бонусы, инициативу, инвентарь как механику. Если мастер пишет «Айра достигла 5 уровня» — агент явно отвечает, что это механика персонажа и в канон она не пойдёт; при необходимости предлагает сохранить сюжетный факт («Айра прошла обряд посвящения», если из текста ясно).

Поля под механику в `CanonCharacter` не закладываются: ни `level`, ни `class`, ни `hp`, ни `inventory_items`. Item с `owner_character_id` — это **исключение по смыслу** (артефакт = лорный объект, не «инвентарь»).

**Q8. Visibility — общее решение.** На Character `visibility: Visibility = "public"`. «Истинное имя лича — Малдрен» = Character с `visibility=gm_only` (или отдельный Character + relation `same_as`, когда дойдём до Сценария 2). Никаких per-field `secret_facts`; скрытые факты идут через Source с `visibility=gm_only`.

#### Item / Artifact — концептуальная модель _(закрыто 2026-05-13)_

**Определение.** Item — это **именованный лорный объект мира**: физическая вещь, у которой есть собственное имя и сюжетное/историческое значение. Меч Первых, Кольцо Теней, Амулет Луны, Печать Первых, ключ от северных ворот крепости Брекк. Item существует в мире независимо от того, у кого он сейчас в руках.

Item в Lorekeeper — **не инвентарь и не игровая механика**. Это лорный артефакт или именованный предмет, который мастер ведёт как часть истории, а не как +1 к атаке.

**Задачи мастера, которые решает Item:**

1. **Реестр артефактов и реликвий** — у каждого своя история, своя сила, своё место в мире.
2. **Двигатель сюжета** — найти / вернуть / уничтожить / спрятать. Артефакт = квест.
3. **Связь между сущностями через предмет** — «Х подарил Y предмет Z», «партия нашла в логове Барона Печать Первых».
4. **Носитель тайн** — предмет как ключ к загадке («истинная сила Меча Первых известна только посвящённым»).
5. **Якорь для локаций и фракций** — «Меч хранится в храме Огня» = причина существования храма в каноне; «реликвия рода» = причина существования рода.
6. **Хронология находок и потерь** — что партия собрала, что потеряла, где осели реликвии после развала династии.

**Правила «что считать Item»:**

- ✅ Именованный конкретный экземпляр с лорным значением.
- ❌ Безымянная вещь («какой-то старый меч», «пара монет», «несколько свитков»). Зеркалит правило «нет имени — нет сущности».
- ❌ Инвентарь персонажа как механика (3 факела, 50 рационов, +1 к атаке).
- ❌ Категория вещей («мечи», «зелья», «золото»). Категория ≠ сущность.
- ❌ Массовый расходник, даже именованный («зелье лечения» вообще). Item появляется только когда **конкретный** экземпляр имеет лорное значение.
- ❌ Часть характеристик персонажа («Кейра — мечник» — это `key_facts` персонажа, а не Item).

**Item vs Artifact (`kind`):**

- **`artifact`** — уникальный, со значимым лором: магия и/или исторический вес и/или сюжетная привязка. Меч Первых, Кольцо Теней, Амулет Луны, Печать Первых.
- **`item`** — именованный, но обычный: ключ от северных ворот, печать гильдии, личный медальон NPC. Может фигурировать в сюжете, но сам по себе чудес не творит.

Граница серая, но enum сохраняем. Калибровка — задача extraction-промпта.

**Q1. Owner — кто может «владеть» предметом.** Четыре независимых поля:

- `owner_character_id: str | None` — конкретный персонаж носит/держит.
- `owner_faction_id: str | None` — фракция-хранитель (Меч в Орденских хранилищах).
- `held_by_party: bool` — партия таскает с собой. Зеркало `party_involved` у Event: партия как абстракция не Character и не Faction, но как агент действует.
- `current_location_id: str | None` — где предмет физически находится.

Поля независимы, могут заполняться в любых сочетаниях. Если все пустые → предмет «потерян/неизвестно где», обычно вместе с `status=lost`.

**Правило приоритета для UI карточки:** «где предмет сейчас» = `current_location_id` (явно) → иначе `owner_character.current_location_id` → иначе `owner_faction.headquarters_location_id` → иначе «местоположение неизвестно». Это правило отображения, не схемы.

**Q2. Origin — где предмет появился.** Одно поле:

- `origin_event_id: str | None` — событие появления (создание, находка, появление в мире).

Создатель, эпоха, место создания — всё хранится **на самом event'е** (его `participant_character_ids`, `location_ids`, `era_label`, `kind=lore`). Дублирующих полей `created_by_character_id` / `forged_at_location_id` / `era` на Item не делаем. Один источник правды — лор-event, остальное — обратный lookup.

Событие появления — это **классный лорный event**: «Меч Первых выкован дварфским кузнецом Тораном в эпоху Древних в кузне горы Карн-Дрор», `kind=lore`, в нём вся история создания.

**Q3. Subtype / категория — НЕ добавляем.** Свободного текста (типа `weapon`, `amulet`, `book`) на Item нет. Если нужно — фраза в `description` / `key_facts`. Это намеренно двигает Item подальше от charsheet-логики.

**Q4. Status — оставляем существующий набор.**

```
ItemStatus = "active" | "lost" | "destroyed" | "hidden" | "unknown"
```

- `active` — у предмета известный владелец / локация, всё нормально.
- `lost` — потерян, никто не знает где (Амулет Луны после бегства из Финемера).
- `destroyed` — уничтожен.
- `hidden` — намеренно спрятан, мастер знает где.
- `unknown` — статус неизвестен.

Все пять покрывают кейсы Iter 1. Расширим только если в Сценарии 4 окажется тесно.

**История владения — через Event, не через поля.** Передача владельца = event с `item_ids` + смена `owner_*` на самом Item. Timeline предмета = обратный lookup по `Event.item_ids`. Поле `previous_owners` НЕ добавляем.

**Магические свойства — через key_facts / description.** Никаких `magic_properties: list`. Артефакт «горит огнём против нежити» — это `key_facts`, лор, не механика. Никаких `weight`, `cost`, `material`, `damage` — Lorekeeper не charsheet.

**Множественность экземпляров.** Каждый именованный экземпляр = свой Item. Две копии Книги Теней разошлись в разные руки → два Item. Массовый «зелье лечения» — не Item вообще. Полей `copies_count` / `instance_of_item_id` нет.

**Visibility (общее решение).** На Item `visibility: Visibility = "public"`. «Истинная природа артефакта известна только мастеру» — либо вся сущность `gm_only`, либо отдельный gm_only Source с этим фактом. Per-field secret_facts не делаем.

**Связь с Event и StoryThread (уже зафиксировано):** `Event.item_ids`, `StoryThread.related_item_ids`. Один предмет может фигурировать в нескольких событиях и быть «носителем» нескольких сюжетных веток.

#### Location — концептуальная модель _(закрыто 2026-05-13)_

**Определение.** Location — это **место в мире**: физическое или метафизическое пространство, у которого есть собственное имя и границы на воображаемой карте мастера. Диапазон масштабов огромен: от плана бытия (Царство Теней) до комнаты в подвале. От «Морвей» (остров) до «Сломанный Щит» (таверна).

Location — не Faction (территория, а не агент), не Character (не индивид), не Item (не переносится из рук в руки). Это **принимающая сторона** для большинства ссылок: Character привязывается через `current_location_id` / `origin_location_id`, Item — через `current_location_id`, Event — через `location_ids`, Faction — через `headquarters_location_id`.

**Задачи мастера, которые решает Location:**

1. **Карта мира** — куда персонажи могут пойти, что где находится.
2. **Иерархия мест** — таверна в городе, город в регионе, регион на континенте. Структура для подготовки сцены.
3. **Декорация и атмосфера** — мастер описывает место, не вспоминая каждый раз.
4. **Якорь событий** — «битва у моста через Ален», «обряд в храме Огня». Без места события абстрактны.
5. **Контейнер NPC и фракций** — кто здесь живёт, кто контролирует.
6. **Носитель состояния во времени** — Нерис был городом, теперь руины; деревня Ольховка сожжена.
7. **Контейнер артефактов и реликвий** — Меч хранится в храме, Книга в библиотеке.
8. **Социальный контекст** — «эта территория контролируется Y». Без фракций локации — мёртвые места.

**Что НЕ Location:**

- ❌ Безымянное место («ближайшая деревня», «таверна у дороги», «лес неподалёку»). Зеркалит правило «нет имени — нет сущности».
- ❌ Биом без имени («лес», «горы» вообще). Это `terrain_type` других локаций. Локацией становится только когда у леса/гор есть собственное имя («Великий Чернолес», «Сланцевые горы»).
- ❌ Государство **как политическая структура** — это Faction (Корона Варна, Совет Финемера). Государство **как территория** — Location (level=state). Это разделение зафиксировано в Faction-блоке.
- ❌ Фракция, даже если у неё есть штаб. Здание штаба — отдельная Location, со ссылкой `headquarters_location_id` со стороны Faction.
- ❌ Раса/вид без территории («эльфийские земли» как абстракция → не Location).

**Q1. Иерархия плоская.** `parent_location_id` — дерево без жёстких правил «level X может быть только в level Y». В типичных кампаниях полно исключений: вольный город в империи, остров-государство, settlement внутри огромной site-«крепости». Типичные паттерны (settlement содержит site/building, state содержит region/settlement) идут в extraction-промпт как подсказки, но не валидируются.

**Уровни напомнить:**

- `region` — крупная **географическая или историческая** область, часто природного характера. Может пересекать государственные границы. «Северные земли», «Великая пустошь», «Великий Чернолес». Обычно между `state` и `settlement`, но может быть и без state.
- `district` — **внутригородское** деление, район/квартал. Живёт **внутри `settlement`**. «Верхний квартал», «Восточный квартал», «Старый город», «Доки». Не пересекает settlement'ы.

**Q2. Природные объекты с собственным именем — отдельные Location.**

- «Чёрное море» с именем → Location (level=region или site по масштабу, terrain_type=sea). «Морвей» → Location (terrain_type=island, parent_location_id=Чёрное море).
- Безымянный биом → только `terrain_type` соседней Location («лес неподалёку от деревни» — это `terrain_type=forest` у деревни).

Правило: **имя на карте — сущность, без имени — атрибут**. Зеркалит общее «нет имени — нет сущности».

**Q3. `subtype` — свободный текст.** Не enum.

- Аргумент против enum: subtype слишком открыт («таверна / постоялый двор / трактир / корчма» = одно или разное? «шахта / каменоломня / карьер»?). Жёсткий список не покроет.
- Аргумент за свободный текст: extraction-промпт даёт **подсказки по level'у**:
  - для `settlement` — village / town / city / capital;
  - для `site` — dungeon / ruins / cave / grove / shrine / mine / pass;
  - for `building` — tavern / temple / manor / library / barracks;
  - for `district` — residential / market / docks / slums / royal.

Если на тестах окажется хаос — введём enum с fallback `other`. **Отличается** от решения по `FactionKind` (там enum): kind фракции дискретен (cult vs guild vs order — мало пересечений), subtype локации — нет.

**Q4. `subtype` = исходная природа места, `status` = текущее состояние.**

Самая болезненная развилка. Решение — `subtype` хранит **что это такое по своей сути**, `status` отвечает за **в каком оно сейчас состоянии**:

- Нерис → `subtype="город"`, `status="destroyed"`. UI рендерит «Нерис — разрушенный город» или «руины Нериса».
- Ольховка → `subtype="деревня"`, `status="destroyed"`.

«Руины» — это **не природа** места, это его **состояние**. Аналог: мёртвый персонаж — это всё ещё `species=human`, не `species=corpse`. Это правило идёт в extraction-промпт: на тексте «теперь это руины города Нерис» extraction должен записать `subtype="город"` + `status="destroyed"`, не `subtype="руины"`.

**Бонус:** структурный запрос «покажи все разрушенные города» работает (`level=settlement AND subtype=город AND status=destroyed`). Если бы хранили «руины» в subtype — этот запрос невозможен.

**Q5. `controlled_by_faction_id` — кто контролирует территорию.**

- **Где хранится:** на Location. Одно поле, не список (90% случаев — один контроллёр; спорные/исторические — через events).
- **На Faction поля нет.** «Контролируемые территории» — обратный lookup на стороне Faction.
- **Зеркало других решений:** ссылка хранится на «единственной» стороне (территория обычно имеет одного контроллёра), обратный lookup на «множественной» (фракция может контролировать много мест). Тот же принцип, что для Character.affiliation*faction_ids, Faction.leader_character_ids, Item.owner*\*.

**Контроль НЕ наследуется от parent.** Поле — локальное, отвечает за «кто контролирует **именно это** место». Никакого каскада «settlement пустое → берём из state».

Почему:

- Корона Варна контролирует государство, но **внутри** Варна Гильдия Воров фактически держит Доки Корнхейма. Каскад стёр бы этот нюанс.
- Смыслы контроля на разных level'ах разные: государство контролируется политически, таверна — как собственность, подземелье — как захват. Каскад смешивает разные виды отношений.
- Простое правило для extraction: «кто контролирует это место конкретно». Точка.
- Зеркалит `status` (Корнхейм разрушен ≠ все здания в нём `destroyed`) и `Event.location_ids` (событие в таверне ≠ событие в государстве).

**UI** обходит дерево `parent_location_id` вверх и показывает читаемую **цепочку власти**: «Корнхейм: контроллёр — Совет Корнхейма. _В составе Королевства Варн, которое контролирует Корона Варна._» Это отображение, не каскад в данных. Пустые уровни UI пропускает.

**Открытый вопрос: вражда между государствами** (см. Faction-блок). Поскольку State — это Location, а Location ↔ Location relations у нас нет, война государств моделируется через `relations` между их правящими фракциями (с возможным auto-Faction fallback для каждого state'а). Окончательное решение принимаем на стадии extraction-промпта Iter 2.

**Q6. Население / inhabitants — НЕ добавляем.**

Никаких `population_label`, `inhabitants_summary`, `population_count`. Двигает Lorekeeper подальше от «бухгалтерии мира».

- Для settlement масштаб уже передаёт `subtype` («деревня» / «город» / «столица»).
- Для site/building обычно не релевантно.

**Правило для extraction (идёт в промпт Iter 2):** точные демографические/экономические/торговые факты («Финемер: ~12 тыс жителей», «город живёт за счёт серебряных рудников», «крупный торговый узел») идут в `key_facts`, а не в отдельные поля. То же касается точных географических координат и расстояний.

**Q7. Геопозиция — НЕ добавляем.**

Никаких `coordinates_hint`, `map_position`, `grid_x/y`, `bearing`. Свободный текст «к северу от Корнхейма», «в трёх днях пути от моря» — в `description` / `key_facts`. Карта в Lorekeeper появится в Сценарии 5+; тогда станет понятно, какое именно поле нужно (координаты? grid? bearing?), и расширим осознанно.

**Q8. Связи Location ↔ Location (кроме parent) — через `key_facts`, не структурное поле.**

Дороги, тропы, порталы, тоннели, морские пути, секретные проходы — на MVP свободным текстом в `key_facts` соответствующих локаций. Структурное `connections: list[LocationConnection]` — post-MVP.

Почему так:

- На MVP агент путей не строит. «Северный Тракт ведёт из Корнхейма в Финемер, 3 дня, опасен гоблинами» в `key_facts` Корнхейма решает задачу подготовки сцены полностью.
- На MVP неизвестно, какие именно поля нужны в `LocationConnection` (`kind`? `travel_time`? `is_bidirectional`? `hazards`? `via_terrain`?). Угадывать сейчас преждевременно — лучше дождаться карты.
- Уже есть «бесплатный» граф через Events: перемещения партии (`Event.location_ids=[A, B, C]`) реконструируют реальные маршруты. Аналог решения по timeline сущности и причинности — вычисляем из истории, не дублируем.
- Если связь сама — лорная сущность («Древний Тоннель соединяет храм с гробницей, в нём гулы»), сам тоннель становится отдельной Location (level=site, subtype=«тоннель»), а упоминания «соединён с X» живут в `key_facts` храма и гробницы.

**Visibility (общее решение).** На Location `visibility: Visibility = "public"`. Тайный храм культа, секретный проход в подземелье, скрытое убежище → `gm_only`. Известный город → `public`. Никаких per-field secret_facts; скрытые факты идут через Source с `visibility=gm_only`.

**Связь Location с другими сущностями (уже зафиксировано):**

- Character: `current_location_id`, `origin_location_id`.
- Item: `current_location_id`, плюс `origin_event_id` ссылается на event, у которого есть `location_ids`.
- Event: `location_ids: list[str]` (несколько мест на событие).
- Faction: `headquarters_location_id`.
- StoryThread: `related_location_ids`.

История изменений локации (status, subtype-переписывания, смены контроля) — через Events. Timeline места = обратный lookup по `Event.location_ids`. Никаких полей `previous_subtype` / `previous_status` / `previous_controlled_by` на Location.
